Foundation Series — F-04

Claude 작업 구조를
이해하는 기본 개념

읽는 시간 약 12분 · 무료 공개 · 다음 → System Series S-01

Claude를 단순히 "질문하면 답하는 도구"로 쓰면 30%도 활용하지 못합니다. Claude가 어떻게 생각하고, 어떤 구조로 작동하는지를 이해하면 훨씬 정밀하게 쓸 수 있습니다. 이 세션은 그 구조를 설명합니다.

4.1
Claude는 어떻게 생각하는가

Claude는 대화창에 입력된 모든 텍스트를 한꺼번에 읽고 답합니다. 이것을 컨텍스트 창(Context Window)이라고 합니다. 대화 내용, 업로드한 파일, 시스템 지시사항 — 이 모든 것이 Claude가 읽는 "맥락"입니다.

컨텍스트 창이란
Claude가 한 번에 처리할 수 있는 텍스트의 양입니다. Claude의 컨텍스트 창은 약 200,000 토큰으로, 일반 소설책 한 권 분량의 텍스트를 한 번에 읽고 이해할 수 있습니다.

중요한 점 — 대화가 길어질수록 앞부분의 내용을 덜 중요하게 처리합니다. 핵심 정보는 항상 명확하게 반복해서 전달하는 게 좋습니다.

4.2
Context Engineering — 맥락을 설계하는 법

좋은 결과물은 좋은 프롬프트가 아니라 좋은 맥락 설계에서 나옵니다. 프롬프트 엔지니어링이 "질문을 잘하는 것"이라면, Context Engineering은 "Claude가 작업하는 환경 전체를 설계하는 것"입니다.

Layer 1
시스템 지시 (Project Instructions)
Claude가 이 프로젝트에서 항상 따르는 기본 규칙. 역할, 말투, 금지 표현 등을 정의합니다.
예: "당신은 채움의 브랜드 AI입니다. 모든 결과물은 BRAND.md 기준을 따르세요."
Layer 2
브랜드 지식 (Knowledge Files)
Projects에 올려둔 파일들. BRAND.md, 제품 설명, 타겟 페르소나 등 Claude가 항상 참고하는 문서입니다.
예: BRAND.md, 제품 카탈로그, 고객 FAQ
Layer 3
대화 맥락 (Conversation History)
현재 대화에서 오간 내용. 앞서 말한 조건, 수정 요청, 승인한 내용이 모두 포함됩니다.
예: "아까 말한 톤으로", "이전 버전에서 첫 문장만 바꿔서"
Layer 4
현재 요청 (User Request)
지금 입력하는 메시지. Layer 1~3이 잘 설계돼 있으면 짧고 간단한 요청만으로도 원하는 결과가 나옵니다.
예: "이번 주 인스타 캡션 3개 써줘" (한 줄로 충분)
4.3
Claude가 작업하는 방식 — Plan → 실행 → 확인

Claude는 복잡한 작업을 받으면 무작정 실행하지 않습니다. 내부적으로 계획을 세우고, 단계별로 실행하고, 불확실한 부분은 확인을 요청합니다.

Plan — 먼저 계획을 세웁니다
큰 작업을 받으면 Claude는 어떻게 접근할지 먼저 생각합니다. "한 달치 콘텐츠 캘린더 만들어줘"라고 하면 — 먼저 채널별로 나누고, 주제를 구성하고, 형식을 정하는 계획을 세웁니다. 이 계획을 먼저 보여달라고 요청하면 방향을 조정할 수 있습니다.
실행 — 단계별로 처리합니다
한 번에 모든 걸 내놓기보다 단계별로 작업합니다. 긴 문서를 분석할 때는 섹션별로, 여러 채널 콘텐츠를 만들 때는 채널별로 나눠서 진행합니다. 중간에 방향을 수정하기 쉬운 구조입니다.
확인 — 모호한 부분은 먼저 물어봅니다
요청이 불명확하거나 중요한 결정이 필요한 경우, Claude는 추측하지 않고 확인을 요청합니다. 이게 귀찮게 느껴질 수 있지만 — 엉뚱한 결과물을 받는 것보다 훨씬 낫습니다. Project Instructions에 "불확실하면 추측하지 말고 질문하세요"라고 명시하면 더 정확하게 작동합니다.
4.4
Claude가 할 수 있는 것과 못하는 것

Claude를 잘 쓰려면 한계도 정확히 알아야 합니다. 과신하면 오히려 브랜드에 해가 됩니다.

4.5
채움이 Claude를 쓰는 구조

위의 개념을 실제 브랜드 운영에 적용하면 이렇게 됩니다.

채움의 Claude 운영 구조
Layer 1 (시스템) — 브랜드별 Project Instructions. 역할, 말투, 금지 표현, 결과물 형식 정의

Layer 2 (지식) — BRAND.md + 제품 설명 + 타겟 페르소나. 3개 파일로 브랜드 뇌 구성

Layer 3 (자동화) — MCP로 Notion 연결. Claude가 직접 콘텐츠 캘린더 읽고 초안 저장

Layer 4 (요청) — "이번 주 콘텐츠 뽑아줘" 한 줄로 전체 프로세스 실행
이 구조가 만드는 변화
레이어가 잘 쌓여 있으면 — 짧은 요청으로 긴 결과물이 나옵니다.
레이어 없이 쓰면 — 매번 긴 설명을 붙여야 하고 결과도 들쭉날쭉합니다.

Foundation에서 배운 것들 (가입 → Projects 세팅 → BRAND.md → MCP 연결)이 모두 이 레이어를 쌓는 과정이었습니다.
결론

Claude는 맥락을 먹고 자랍니다. 좋은 맥락을 설계할수록 요청은 짧아지고 결과물은 정확해집니다.
Foundation Series는 여기까지입니다. 이제 System Series에서 이 구조를 내 브랜드에 직접 적용합니다.
